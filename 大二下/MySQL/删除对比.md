# TRUNCATE & DELETE

| 对比项                     | `TRUNCATE`                              | `DELETE` (无WHERE)               |
| :---------------------- | :-------------------------------------- | :------------------------------ |
| **本质**                  | DDL（数据定义语言）                             | DML（数据操作语言）                     |
| **执行速度**                | **极快**（重建表）                             | **慢**（逐行删，记录每一行删除日志）            |
| **能否回滚**                | **通常不能回滚**（事务中一般不可恢复，部分数据库如PG可以但需要特殊处理） | **可以回滚**（只要在事务内，`ROLLBACK` 可恢复） |
| **自增列（AUTO_INCREMENT）** | **重置**为初始值（通常从1开始）                      | **不重置**，继续在上次最大值基础上自增           |
| **触发器**                 | 不触发 `DELETE` 触发器                        | 触发每行的 `DELETE` 触发器              |
| **锁**                   | 通常加**表级锁**（或更高级锁）                       | 逐行加**行锁**（全表删时可能自动升级为表锁）        |
| **磁盘空间**                | **立即释放**给操作系统，表文件变小                     | 只是标记空间为“可复用”，表文件大小**不变**        |
| **权限**                  | 需要更高权限（如 `ALTER` 权限）                    | 只需要 `DELETE` 权限                 |

> 总结: **`TRUNCATE` 是“物理清空并重建表”，速度快、不可回滚、重置自增ID；  
> `DELETE` 是“逻辑逐行删除”，速度慢、可回滚、不重置自增ID。**

# TRUNCATE & DROP
## 一、数据内容操作对比（表中数据的处理）

| 对比点           | TRUNCATE                | DROP                               |
| :------------ | :---------------------- | :--------------------------------- |
| **数据是否保留**    | **删除全部数据**（数据被清空）       | **删除全部数据**（数据被删除）                  |
| **删除后能否恢复数据** | 通常**不可回滚**（部分数据库在事务中例外） | 通常**不可回滚**                         |
| **数据删除方式**    | 先删除表，再重建一个**相同结构**的空表   | 直接删除表的**定义+数据**                    |
| **删除速度**      | **非常快**（不逐行删）           | **非常快**（同样不逐行删，比TRUNCATE稍快，因为不用重建） |

> 结论：**从数据内容角度看，两者都会清空数据，且都不可随便回滚。** 区别在于 `TRUNCATE` 会保留一个“空壳表”让数据消失；`DROP` 则连数据带“壳”都消失。

## 二、数据表操作对比（表结构/元数据的处理）

| 对比点             | TRUNCATE                              | DROP                                |
| :-------------- | :------------------------------------ | :---------------------------------- |
| **表结构是否存在**     | **依然存在**（空表）                          | **完全删除**（表不存在）                      |
| **列、约束、索引**     | **保留**所有定义                            | **删除**所有定义                          |
| **触发器**         | 保留（但DML触发器不触发）                        | 触发器被删除                              |
| **权限**          | 表还在，权限**保留**                          | 表被删，权限**清除**                        |
| **外键依赖**        | 如果有外键引用该表，`TRUNCATE` **可能失败**（取决于数据库） | `DROP` 表时如果有外键引用，通常**也会失败**（除非级联删除） |
| **操作后能否立即插入数据** | **可以**（直接 `INSERT`）                   | 必须先重新 `CREATE TABLE` 才能插数据          |
> 结论：**从数据表操作角度看，`TRUNCATE` 只清空；`DROP` 是删除。**

# 补充：与 `DELETE` 的简单对比

| 命令         | 数据内容       | 表结构 | 自增ID | 速度  | 可回滚  |
| :--------- | :--------- | :-- | :--- | :-- | :--- |
| `DELETE`   | 删除指定行（或全部） | 保留  | 不重置  | 慢   | 可以   |
| `TRUNCATE` | 删除全部行      | 保留  | 重置   | 快   | 一般不可 |
| `DROP`     | 删除全部行      | 删除  | 无    | 最快  | 一般不可 |
